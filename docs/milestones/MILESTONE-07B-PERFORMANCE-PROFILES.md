# 目标 7B：性能档位与播放运行时优化

完成日期：2026-07-16

## 用户功能

- 顶部“性能”入口最终收敛为自动、流畅、高清三个用户选项；均衡保留为自动档内部状态和固定基准参数。
- 自动档按设备能力选择内部有效档位；当前 M5 Pro 自动采用均衡。
- 档位统一控制主视口、监看小窗和坐标 Gizmo 的 DPR、抗锯齿及 UI 播放刷新频率。
- 720p/1080p 参考视频导出使用明确的导出尺寸，不被编辑预览档位降清晰度。
- 用户选择会持久化；标准基准也可用 `performance=fluid|balanced|quality` 固定测试档位。

## 结构性性能修复

7A 基准证明，单纯降低 DPR 只能把约 4.4 FPS 提高到约 4.6 FPS。主要瓶颈是播放进度每帧写入 Zustand，导致 React 和多棵 R3F 场景重复渲染。

本阶段将高频播放进度移入独立 runtime：

- 每帧只更新 runtime，并按档位频率更新底部时间和其他普通 UI。
- 相机、运动人物、运动道具和跟踪目标直接读取 runtime 进度。
- UE4、Mixamo、外部 FBX/GLB 动画和基础人偶均保留逐帧动作。
- 时间轴拖动、轨迹点选择、撤销、项目替换等 store 入口统一同步 runtime。
- 暂停时将 runtime 的最终进度回写 store，因此暂停不会回到起点或退回上一次 UI 刷新位置。
- 相同进度不重复通知运行时对象。

## 同一压力场景复测

测试环境与 7A 相同：Apple M5 Pro、DPR 2、1280 x 650 CSS 视口、25 个动作人物、12 个道具、三点循环镜头、主视口和成片监看。

| 档位 | 主 Canvas 实际像素 | 平均 FPS | 平均帧时间 | P95 | 1% low |
| --- | ---: | ---: | ---: | ---: | ---: |
| 流畅 | 960 x 487 | 96.7 | 10.34 ms | 16.7 ms | 59.9 FPS |
| 均衡 | 1280 x 650 | 96.3 | 10.38 ms | 16.9 ms | 57.1 FPS |
| 画质优先 | 2560 x 1300 | 92.7 | 10.79 ms | 16.9 ms | 57.8 FPS |

对比 7A 的画质基线约 4.4 FPS / P95 233.6 ms，画质优先档在不降低主 Canvas 像素尺寸的情况下达到约 92.7 FPS / P95 16.9 ms。三档 draw calls 均为 37，三角形约 271,194，压力场景内容没有删减。

浏览器的 requestAnimationFrame 在高刷新率屏幕上可超过 60 FPS，因此平均 FPS 高于 60；1% low 和 P95 更接近持续流畅性的保守指标。
